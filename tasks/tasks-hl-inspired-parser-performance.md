    **Root cause** (verified by the Stage B utilisation collapse from 84.6 % → 10.4 %): the streaming pipeline performs **at least two element-wise moves per `LogLine`** in any non-trivial sink — (1) `parsed.lines.push_back(std::move(logLine))` during Stage B, and (2) `BufferingSink::OnBatch` → `mLines.insert(end, make_move_iterator(begin), make_move_iterator(end))` (or the equivalent `pending.lines.insert(...)` for coalesced sinks at `json_parser.cpp:1386`). `std::vector::insert` from move-iterators is element-wise, not a buffer splice. Adding a 16-slot inline buffer of `pair<KeyId, LogValue>` (~40 B each) plus the `std::vector<…> mHeap` fallback inside `SmallVector` blows `sizeof(LogLine)` from ~96 B → ~830 B. Per-line memcpy traffic for 1 M lines × 2 moves goes from ~200 MB → ~1.66 GB, which both costs wall-time directly and saturates the memory bus, starving Stage B workers. LLVM-style "indirect" `SmallVector` does not help: an inline buffer is part of the source object, so it can never be stolen on move; only heap-mode moves are pointer-swap, and most lines in `[large]` (5 fields) and `[cancellation]` (small) stay inline. There is no N where this design wins under the current pipeline.

    **Why §4.1's `[get_value_micro]` was supposed to win and didn't.** §4.1 was framed as a wash-or-win on `[get_value_micro]` and a clear win on the streaming benchmarks because the alloc-savings (PRD G2: ≤ 0.1 alloc/line) would dominate. Empirically the alloc-savings *did* materialise in the prototype's `[allocations]` printout, but they did not pay back the per-line move-traffic cost; the alloc-savings would have been ≈ 1 alloc × ~50 ns × 1 M lines = ~50 ms, while the extra memcpy traffic is ~1.5 GB × ~10 GB/s = ~150 ms. The ratio is fundamentally backwards for this codebase's pipeline.

    **Future redesign options** (deferred — none small enough for this PRD):
    - Per-thread arena / `BufFactory` style backing for `mValues`'s heap buffer (move = pointer-swap, alloc = O(1) per batch). Closest to PRD §4.3's batch-pool work but at the inner-vector level.
    - `unique_ptr<LogLine>` ownership through the pipeline (move = pointer-swap, but reintroduces the per-line heap alloc §4.1 was trying to avoid).
    - Splice-style `std::vector<LogLine>` accumulator that swaps capacity instead of `insert`-ing; would require a custom container in `BufferingSink::mLines`.

    **Disposition.** Per PRD G6 ("revert if a task does not produce a measurable improvement"), revert. The prototype (`small_vector.hpp`, `test_small_vector.cpp`, `LogLine` integration, `JsonParser::ParseLine` return-type change, `[allocations]` printout addition) was deleted from the working tree before this commit; baseline-restoration was verified via the same benchmark sweep above. M1 trajectory after this commit: still 1138 MB/s post-1.0 (the IPO/LTO win); aspirational ≥ 1 GB/s is already met by §4.0 alone, so §4.1's planned aspirational lift is forfeit but not load-bearing.

    **Root cause** (verified by inspecting the linked `loglib.lib` symbols with `llvm-nm`): clang-cl emits `XXH_INLINE_XXH3_64bits` as an out-of-line static function despite `XXH_INLINE_ALL=1`, so `TransparentStringHash::operator()` does a function-call hop into the entry-point dispatcher (length-based switch into the 4–8 / 9–16 / 17–128 byte sub-paths). For our 5–20-byte field-name keys, the function-call overhead plus the dispatch is comparable to the per-byte loop in MSVC STL's `std::hash<std::string_view>` (which fully inlines into the call site), so the two end up roughly equivalent in steady state. xxHash's published 10 GB/s number is for fully-inlined, large-input, vectorised paths; the small-key, dispatch-included path that our `KeyIndex::Find` actually exercises is much closer to std-hash territory on this codebase.

    **Why §4.2's gate prediction did not pan out.** 3.2's gate input — `[no_thread_local_cache]` MB/s 73 % below `[large]` MB/s — was correctly flagging that *something* in the canonical `KeyIndex::Find` path is expensive when the per-worker absorber (parser-perf parent task 1.0) is disabled. That gate diagnosed the cost as the hash function. The empirical sweep at 4.7 disproved that diagnosis: swapping `std::hash` → xxh3-64 across both `TransparentStringHash::operator()` and `Impl::ShardIndex` did not measurably move `[no_thread_local_cache]`'s warm-up MB/s (286–302 across three runs vs. 289 baseline; mean of 5 within ~3 % of baseline). The dominant cost of the canonical-path `Find` is therefore the per-shard `std::shared_lock<std::shared_mutex>` acquire/release, not the hash. That is consistent with the pre-PRD architecture: 16 shards × 1 shared_lock per `Find` × 1 M lines on `[no_thread_local_cache]` is ~16 M atomic CAS operations, which dominates a ~5 ns hash by an order of magnitude.

    **Disposition.** Per PRD G6 ("revert if a task does not produce a measurable improvement"), revert. The prototype (`xxhash` `FetchContent` block, the `xxhash_inline` `INTERFACE` target plus the `xxHash::xxhash` alias, both `TransparentStringHash::operator()` swap sites, the `Impl::ShardIndex` swap, the `[key_index][hash_distribution]` test, and the `xxHash::xxhash` link entry on both `loglib` and `tests`) was deleted from the working tree before this commit; baseline-restoration was verified via `git diff` (empty post-revert) and a re-run of `ctest --preset local` (95/95 passing).     **Future redesign options for the gate-input signal** (deferred — out of scope for this PRD):
    - Replace `std::shared_mutex` per-shard locks with a `std::atomic<uint64_t>` insert-counter + a lock-free read path that publishes through a per-shard `tsl::robin_map<...>` snapshot pointer (read-mostly workloads pay zero lock-acquire cost).
    - Make the per-worker absorber a *first-class* part of the public `KeyIndex` API instead of a parser-internal detail, so any consumer (including the GUI thread's `RefreshColumnKeyIdsForKeys`) can opt into it without re-wiring through `WorkerState`.
    - Drop the shard count to 1 and replace the per-shard `tsl::robin_map` with a single `concurrent_hash_map` (oneTBB) whose internal sharding is already SIMD-tuned; would require validating cancellation-latency parity since `concurrent_hash_map` doesn't expose `shared_lock`-style readers.
