    **Root cause** (verified by the Stage B utilisation collapse from 84.6 % → 10.4 %): the streaming pipeline performs **at least two element-wise moves per `LogLine`** in any non-trivial sink — (1) `parsed.lines.push_back(std::move(logLine))` during Stage B, and (2) `BufferingSink::OnBatch` → `mLines.insert(end, make_move_iterator(begin), make_move_iterator(end))` (or the equivalent `pending.lines.insert(...)` for coalesced sinks at `json_parser.cpp:1386`). `std::vector::insert` from move-iterators is element-wise, not a buffer splice. Adding a 16-slot inline buffer of `pair<KeyId, LogValue>` (~40 B each) plus the `std::vector<…> mHeap` fallback inside `SmallVector` blows `sizeof(LogLine)` from ~96 B → ~830 B. Per-line memcpy traffic for 1 M lines × 2 moves goes from ~200 MB → ~1.66 GB, which both costs wall-time directly and saturates the memory bus, starving Stage B workers. LLVM-style "indirect" `SmallVector` does not help: an inline buffer is part of the source object, so it can never be stolen on move; only heap-mode moves are pointer-swap, and most lines in `[large]` (5 fields) and `[cancellation]` (small) stay inline. There is no N where this design wins under the current pipeline.

    **Why §4.1's `[get_value_micro]` was supposed to win and didn't.** §4.1 was framed as a wash-or-win on `[get_value_micro]` and a clear win on the streaming benchmarks because the alloc-savings (PRD G2: ≤ 0.1 alloc/line) would dominate. Empirically the alloc-savings *did* materialise in the prototype's `[allocations]` printout, but they did not pay back the per-line move-traffic cost; the alloc-savings would have been ≈ 1 alloc × ~50 ns × 1 M lines = ~50 ms, while the extra memcpy traffic is ~1.5 GB × ~10 GB/s = ~150 ms. The ratio is fundamentally backwards for this codebase's pipeline.

    **Future redesign options** (deferred — none small enough for this PRD):
    - Per-thread arena / `BufFactory` style backing for `mValues`'s heap buffer (move = pointer-swap, alloc = O(1) per batch). Closest to PRD §4.3's batch-pool work but at the inner-vector level.
    - `unique_ptr<LogLine>` ownership through the pipeline (move = pointer-swap, but reintroduces the per-line heap alloc §4.1 was trying to avoid).
    - Splice-style `std::vector<LogLine>` accumulator that swaps capacity instead of `insert`-ing; would require a custom container in `BufferingSink::mLines`.

    **Disposition.** Per PRD G6 ("revert if a task does not produce a measurable improvement"), revert. The prototype (`small_vector.hpp`, `test_small_vector.cpp`, `LogLine` integration, `JsonParser::ParseLine` return-type change, `[allocations]` printout addition) was deleted from the working tree before this commit; baseline-restoration was verified via the same benchmark sweep above. M1 trajectory after this commit: still 1138 MB/s post-1.0 (the IPO/LTO win); aspirational ≥ 1 GB/s is already met by §4.0 alone, so §4.1's planned aspirational lift is forfeit but not load-bearing.
