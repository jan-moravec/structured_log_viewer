- The static-open path goes through an mmap'd `LogFile` parsed by `JsonParser::ParseStreaming` over the TBB pipeline in `library/src/parser_pipeline.cpp`. Stream Mode reuses everything **above** the source layer (`KeyIndex`, `LogConfiguration` / `LogConfigurationManager`, `LogTable` `BeginStreaming` / `AppendBatch` / `EndStreaming`, `StreamingLogSink`, `WorkerScratchBase` / `BuildTimeColumnSpecs`) and replaces only the byte producer below it.

  - [x] 3.4 The pimpl owns a single dedicated worker `std::thread`. The worker pulls bytes from the native handle into a 64 KiB scratch (size configurable via `Options::readChunkBytes`), takes the impl's mutex briefly to merge the new bytes into a `std::string mPartialLine`, splits at the last `\n` into `std::vector<char> mReadyBuffer`, and notifies `mCv`. `Read(std::span<char>)` drains `mReadyBuffer + mReadyConsumed`; `WaitForBytes(timeout)` parks on the same `mCv` and wakes on bytes-available / Stop / worker-exited.
  - [x] 3.5 Pre-fill walks `[file_size, 0]` in `Options::prefillChunkBytes` (default 64 KiB) slices, counting `\n`s backwards until `N+1` are seen or `pos == 0`. The byte after the `(N+1)`-th newline from the end is the start of the last `N` complete lines; we then read `[lineStart, file_size)` forwards and feed it through the same `AppendBytesAndSplitLocked` path the worker uses, so any trailing partial line naturally seeds `mPartialLine` (PRD 4.1.5.i, 4.1.6). The tail read offset is set to `file_size` at pre-fill time so no byte is read twice.
  - [x] 3.6 Watcher integration is via a private `TailingFileWatcherListener final : public efsw::FileWatchListener`. The constructor calls `addWatch(parent_path, listener, /*recursive=*/false)` (Linux `inotify` and macOS `FSEvents` work at directory granularity per PRD 4.8.5) and filters events down to `mPath.filename()`. On any matching event the listener wakes `mCv`. The 25 ms / 250 ms polling tick (configurable via `Options::pollInterval`) re-evaluates the rotation branches and reads new bytes regardless of watcher delivery.
  - [x] 3.7 Rotation branches in `DetectAndRecoverRotation` evaluate in PRD order (i identity / ii path-missing / iii size-shrunk-on-handle); the first match wins. Branch (i) compares `FromPath(mPath)` against `FromOpenHandle(mHandle)` — when they disagree we close the old handle, open the new path, seek 0, sample new identity. Branch (ii) sets a `mWaiting` flag and keeps the old handle open (POSIX semantics keep its content readable; Windows: the same producer-friendly share flags allow it). Branch (iii) compares `SizeOfOpenHandle()` against `mReadOffset`; on shrink we seek the existing handle to 0 (no re-open per PRD 4.8.6.iii).
  - [x] 3.8 Every rotation: clears `mPartialLine` under the mutex (PRD 4.8.7.i), applies the recovery, leaves `mReadyBuffer` and already-emitted bytes intact, increments `mRotationCount`, and calls `FireRotationCallbackIfDebounced`. The internal `mRotationCount` always increments — it's the test hook that proves the worker actually observed each rotation, separately from the user-facing debounce.
  - [x] 3.9 `Stop()` is the only way the worker exits. It flips `mStopRequested`, locks the mutex briefly to set `mWatcherEventPending = true`, and notifies `mCv`.
